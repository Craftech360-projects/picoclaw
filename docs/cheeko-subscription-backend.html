Data model

DB schema — 4 new tables

Prisma models on the existing Supabase Postgres. The spine is keyed by mac_address, not a FK to ai_device — unbind deletes that row (device.service.js:184) and subscription state must survive it.

subscription_plans · catalog, DB-driven

  • id PK
  • tier 'starter'|'family'|'premium' UNIQUE
  • name · price_inr
  • monthly_question_limit · daily_question_limit
  • daily_minutes_limit (invisible backstop)
  • monthly_image_limit (null=∞) · daily_image_limit
  • features JSON (memory, kid_profiles…)
  • razorpay_plan_id · is_active · timestamps

device_subscriptions · the spine

  • id PK · mac_address UNIQUE
  • status 'trial'|'active'|'grace'|'lapsed'|'cancelled'
  • plan_id → subscription_plans (Family during trial)
  • user_id (payer; survives unbind)
  • trial_started_at · trial_ends_at · trial_used (permanent)
  • billing_cycle 'monthly'
  • current_period_start / end (bucket anchor)
  • grace_until · cancel_at_period_end
  • razorpay_customer_id · razorpay_subscription_id

subscription_events · webhook ledger

  • id PK
  • razorpay_event_id UNIQUE ← idempotency key
  • event_type · mac_address
  • razorpay_subscription_id
  • payload JSON · processed_at · created_at

subscription_admin_audit

  • id PK · admin_user
  • action 'comp_extend'|'trial_regrant'|'plan_override'
  • mac_address · reason
  • before_state JSON · after_state JSON · created_at
Usage is computed, not stored: monthly questions = SUM(message_count) over existing device_token_usage_session WHERE created_at ≥ current_period_start; daily = same over the IST day; images from the imagine-upload path. No counter columns at <100 devices — add a rollup only if the verdict query slows. Phase 2's per-device limit fields are subsumed: limits always come off the plan row.

Edge cases & failure modes

The corners that bite

Severity = how bad it goes if unhandled. P0 = money or child-facing breakage · P1 = wrong state, self-healing possible · P2 = cosmetic/ops.

P0Resold / re-gifted toy with a live mandate

New parent binds a device whose subscription is active and paid by the previous owner — old payer keeps getting charged. On bind where user_id ≠ payer: keep the sub running (kid keeps AI), notify the old payer with a one-tap cancel link, and show "paid by previous owner" in the new parent's app. Never auto-cancel — gifting within a family is the common case.

P0Webhooks silently dead >24h

Razorpay retries for 24h then auto-disables the webhook. A charge that succeeded while dead = parent paid, toy lapsed. Fix: nightly reconciliation job — poll every active|grace|pending subscription against the Razorpay API and repair status; alert on any drift found.

P0Trial-expiry cron dies

If lapse only happened via the daily job, a dead cron = free unlimited AI. The verdict must compute expiry lazily: status='trial' AND now() > trial_ends_at ⇒ treat as lapsed (and repair the row). Cron is just the notifier, never the enforcer.

P1Customer pauses the mandate from their UPI app

subscription.paused arrives — and per Razorpay, only the customer can resume it. Map paused ⇒ GRACE behavior (3-day timer + "resume in your UPI app" push); unresumed ⇒ LAPSED. Don't try to resume server-side; it will fail.

P1Webhook races the checkout response

subscription.activated can land before the portal's checkout POST returns. Handler must upsert by razorpay_subscription_id even when the local row still says trial/lapsed — never assume checkout wrote first.

P1subscription.completed (total_count exhausted)

A "monthly" Razorpay sub has a finite cycle count. Seed total_count high (e.g. 120 cycles = 10 years) and treat completed like cancelled-at-period-end with a re-subscribe push — never let it surprise you at month 12.

P1Refund issued for the current period

Support refunds in the Razorpay dashboard → refund.processed. Full refund of the running period ⇒ set LAPSED immediately (they got the money back); partial/goodwill refunds change nothing. Log to the ledger either way.

P1Account deletion with live subscriptions

DELETE /api/mobile/account must enumerate the parent's device subscriptions and cancel the Razorpay subs first — an orphaned mandate that keeps charging a deleted account is a chargeback machine.

P1Plan change mid-flight (UPI)

Cancel-old + create-new leaves a window where the new mandate is authenticated but not yet charged. Keep the row on the old plan until activated for the new sub arrives; if the parent abandons mandate approval, nothing changed — old plan simply ran out its period.

P2Clock edges

Daily caps roll at IST midnight; period boundaries come from Razorpay timestamps (UTC) — convert once at write, compare in UTC. A session spanning midnight counts toward the day it started.

P2Admin re-grant vs trial_used

trial_used is permanent by policy; the audited admin re-grant sets a fresh trial_ends_at without clearing the flag — the flag means "no automatic trial ever again", not "no trial ever".

P2Verdict during the migration window

Between schema deploy and fleet seeding, bound devices have no subscription row. Verdict must treat missing-row as allowed while ENFORCEMENT_ENABLED=false, and as lapsed once live — the seed script must run before the flag flips.
